⏺ Your five points are solid instincts, but a few are underspecified in ways an
  interviewer will probe. Going through them:

  1. Streaming the CSV isn't the whole memory story. Your dedup/resume mechanism
  needs the same discipline. If "resume without double-sending" means loading a
  sent-log into an in-memory Set<email>, that's a second O(n) structure hiding
  behind the "I streamed the input" claim — ~1M email strings is tens of MB,
  probably fine, but it's inconsistent with your own stated principle. Cheaper
  and more honest: track by row index/offset, not by email string, and rebuild a
  Set<number> (or even a bitset) on startup.

  2. Bounded concurrency ≠ rate limiting. These solve different problems.
  Concurrency caps how many requests are in flight at once (protects your own
  memory/file descriptors); rate limiting caps requests/sec (respects the
  provider's actual constraint). If each mock call takes 200ms and you run 50
  concurrent, you're doing ~250/sec regardless of what the real API allows — you
  haven't modeled the provider's limit at all, just your own parallelism. A
  real provider gives you 429s + Retry-After; if your mock doesn't simulate
  throttling and backoff, your throughput number is fiction. This is the gap I'd
  fix first — it's the difference between "I built a fast script" and "I built
  something that survives contact with a real ESP."

  3. Crash-resume needs a harder definition of "crash." Graceful shutdown
  (SIGINT, drain queue, close file) is easy and proves nothing. Test with
  SIGKILL mid-batch. That surfaces the real question you haven't answered yet:
  do you mark "sent" before dispatch or after provider ack? After ack is
  correct, but then a kill between "provider accepted" and "log write flushed"
  is an unavoidable double-send window — meaning your system is at-least-once,
  not exactly-once. Say that explicitly in the README as a conscious tradeoff
  rather than letting the test appear to "prove" exactly-once when it can't.

  4. You need a third bucket beyond sent/pending: failed. Not building bounce
  handling is a reasonable scope cut, but without a distinct failure log, a
  permanently-bad address (mock 550) looks identical to "never attempted" on
  resume and gets retried forever, or silently swallowed. Separate retryable
  (transient, backoff) from terminal (log and skip) failures — even a fake
  distinction in the mock is more realistic than assuming every send eventually
  succeeds.

  5. The throughput number from the mock is only meaningful if you say what it 
  represents. "1M sends in 40s" against an in-process mock with no real
  network/provider limit is not a claim about real-world performance — it's a
  claim about your event loop's I/O-bound concurrency ceiling. Make the README
  explicit: measured mock throughput X, and separately, "at provider rate limit
  of N req/s, wall time = 1M/N regardless of code." Otherwise the number looks
  like it's claiming more than it can.

  Smaller things worth a line in the README even if out of scope for code: CSV
  field quoting (names with commas break naive split(',')), and not funneling 1M
  individual completion writes through unbuffered per-line fs.appendFile calls
  (batch or use a single write stream, or your "fast" script bottlenecks on disk
  I/O you introduced yourself).

  Net: your framing nails memory and idempotency but treats "fast" and "safe" as
  solved by concurrency + a log file. The gap is rate-limiting/backoff as a
  distinct concern from concurrency, and being upfront about at-least-once
  semantics rather than implying exactly-once.

⏺ Sender pipeline (dry-run + mock) is in. One deliberate deviation from the plan
  worth flagging: I didn't use p-limit. Reasoning — if the CSV producer isn't
  paced to match the concurrency cap, p-limit's internal queue would hold up to
  ~1M pending closures in memory (one per unprocessed row), which quietly
  violates the "streaming, constant memory" commitment the plan is built on.
  Instead send.ts uses a Set-capped in-flight pool: the CSV read pauses once
  concurrency rows are in flight, so memory never scales with file size. Rate
  limiting is still exactly as planned — a per-second window counter in
  limits.ts, no token-bucket library.

  What's implemented and verified:
  - CSV parsing: real streaming parser (csv-parse), not hand-rolled — reads
  {email, name} per row with a row index.
  - Template: compiled once ({{name}} placeholder split into parts, joined per
  row — no per-row regex).
  - Concurrency: --concurrency, default 50, bounded via the in-flight Set.
  - Rate limit: --rate, default 14 req/s.
  - Mock provider: 20-80ms latency, ~0.2% permanent 550, ~1% transient 429 with
  Retry-After honored, max 3 attempts before a transient failure is logged as
  exhausted-retries.
  - Logs: sent.ndjson (row index only) and failed.ndjson (index, email, reason),
  both loaded into Set<number> on startup to skip already-processed rows.
  - Progress line: sent/failed/skipped/remaining (rate/s), updated every 500ms.
  remaining is accurate because of a fast pre-pass line count (streaming,
  doesn't buffer rows).

  Code review findings, most significant first:

  1. Retries-exhausted rows never get a second chance on resume. A row that hit
  failed.ndjson only because it exhausted 3 retries on a transient 429 gets
  treated identically to a permanent 550 forever after — the resume path (the
  whole point of this tool) will never retry it in a later run, even though the
  failure was transient in nature.
  2. No error listener on the sent/failed log write streams. A write failure
  (disk full, permissions) throws as an uncaught exception rather than going
  through the graceful console.error/exit(1) path used elsewhere — bypasses
  main().catch since it's not a promise rejection.
  3. countDataRows counts raw \n bytes rather than using the CSV parser — a
  quoted field with an embedded literal newline would inflate the progress/total
  numbers (doesn't affect actual send correctness, just the displayed stats).
  4. sentLog.end()/failedLog.end() aren't awaited, unlike generate.ts's
  ws.end(); await finished(ws) — inconsistent, and compounds finding #2.
  5. Fixed-window rate limiter allows momentary ~2x bursts at window boundaries
  — worth knowing if --rate is meant as a hard ceiling against a real provider.
  6. loadCompletedIndices silently dedups via Set.add — a genuine duplicate
  index within one log file (a symptom of exactly the bug class this design
  exists to prevent) passes through unflagged.
  7. parseFlags doesn't validate flag names — a typo like --consurrency is
  silently ignored and falls back to the default, no warning.
  8. Mixing --smtp and mock dry-run against the same log files is a silent
  footgun — there's no marker distinguishing a mock-sent row from a
  real-SMTP-sent row, so a dry-run pass followed by --smtp will skip rows as
  "already sent" that were only ever mock-sent.
